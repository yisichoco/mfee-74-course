## Agent 協作準則（LLM）

降低常見 LLM 寫程式錯誤的行為指引；可與專案專屬說明（如 `AGENTS.md`）一併使用。

**取捨：** 本準則偏向謹慎而非速度；對瑣碎任務請自行判斷。

### 1. 動手前先想清楚

**不要臆測。不要藏著困惑。把取捨攤開來說。**

實作前：

- 明確說明你的假設；不確定就問。
- 若有多種解讀，列出來——不要默默選一種。
- 若有更簡單的做法，要說出來；必要時提出異議。
- 若有不清楚之處，先停下來，點出困惑之處，再發問。

### 2. 簡潔優先

**用最少程式碼解決問題。不做投機性擴充。**

- 不做超出需求的功能。
- 不為只用一次的程式碼抽象。
- 不加入未被要求的「彈性」或「可設定性」。
- 不為不可能發生的情境寫錯誤處理。
- 若寫了 200 行其實 50 行就夠，就重寫。

自問：「資深工程師會不會覺得過度複雜？」若是，就簡化。

### 3. 手術式變更

**只動該動的。只清理自己造成的殘留。**

編輯既有程式碼時：

- 不要「順便」改相鄰程式碼、註解或格式。
- 不要重構沒壞的東西。
- 符合既有風格，即使你會用不同寫法。
- 若發現無關的死碼，提及即可——不要擅自刪除。

若你的變更產生孤兒程式碼：

- 刪除**你的變更**導致未使用的 import／變數／函式。
- 除非被要求，不要刪除既有的死碼。

檢驗標準：每一行變更都應能直接對應到使用者的請求。

### 4. 以目標驅動執行

**定義成功標準。反覆執行直到可驗證。**

把任務轉成可驗證的目標：

- 「加驗證」→「先為無效輸入寫測試，再讓測試通過」
- 「修 bug」→「先寫能重現 bug 的測試，再讓測試通過」
- 「重構 X」→「重構前後測試皆須通過」

多步驟任務請簡述計畫：

```
1. [步驟] → 驗證：[檢查方式]
2. [步驟] → 驗證：[檢查方式]
3. [步驟] → 驗證：[檢查方式]
```

明確的成功標準能讓你獨立迭代；模糊的標準（例如「弄好就好」）會需要不斷追問。

---

**本準則有效的跡象：** diff 中不必要的變更少、因過度複雜而重寫的次數少，且釐清問題發生在實作之前而非犯錯之後。

## 開發規範

### 與 ESLint 的關係

- 本節為撰寫準則，**通常嚴於**專案 ESLint 預設（例如 `@typescript-eslint/no-explicit-any` 可能為 `off`，僅 `warn` 的規則亦同）。
- **不得**因 lint 未報錯而放寬：仍應避免 `any`、捏造 API、以斷言繞過型別檢查。
- 提交前仍須通過專案 `pnpm run lint`（或 `package.json` 內對應指令）。

### 文件與草稿放置規範

- 非執行／部署用文件（需求、設計、TODO、筆記、調查、草稿、RUNBOOK）一律放在工作區 `_docs/` 目錄。
- 各專案根目錄僅保留執行、部署與 agent 所需設定；專案細則以該 repo 的 `AGENTS.md` 為準。
- 專案的 API 文件一律放在 `_docs/_API_vX.X.md`。

### 程式碼風格規範

以專案 `.prettierrc` 為準，若無則參考此預設值。

```json
{
  "semi": false,
  "trailingComma": "es5",
  "singleQuote": true,
  "endOfLine": "auto",
  "overrides": [
    {
      "files": ["*.css", "*.scss", "*.sass", "*.less"],
      "options": {
        "singleQuote": false
      }
    }
  ]
}
```

### TypeScript 與 React

- 避免 `any`，優先使用明確型別。
- 元件 Props 使用 `interface`。
- 互動元件才加 `"use client"`。
- 非頁面元件優先具名匯出。

### 檔案頂部註解規範

- 每個程式碼檔案最上方需加入註解，說明**功能**、**目的**。
- 註解需精簡具體，讓新進開發者在未讀完整檔案前理解責任邊界。
- 建議格式（依語言使用對應註解語法）：
  - 功能：主要提供的能力或行為
  - 目的：為何需要此檔案、解決什麼問題

### Inline Comment 規範

- 只在 **WHY 不明顯**時才寫（隱藏限制、特定 bug workaround、會讓讀者意外的行為）。
- 不解釋 WHAT——好的命名本身就是說明。
- 不記錄 task、PR 編號或呼叫者資訊（屬於 commit message，不屬於程式碼）。
- 每處最多一行；禁止多行 comment block。
- 不要隨意刪除既有的註解。尤其是`// TODO`、`// FIXME`、`// NOTE` 這種註解。

### 命名規範

- 檔名統一 `kebab-case`。
- 同層模組可透過資料夾 `index.ts` 做 re-export，降低深層相對路徑。

#### Prisma 命名規範（如有使用 Prisma）

- 資料表用`@map`為 snake_case 風格
- 資料欄位不用`@map`，就prisma預設的風格就行了

### Import 排序規範

1. 外部套件（`react`、`next`、第三方）
2. 內部絕對路徑（`@/lib`、`@/components`）
3. 相對路徑（`./`、`../`）

各群組之間空一行。

### 純函式優先原則

- 共用邏輯維持**純函式**，避免在核心轉換層綁死 Node-only API。
- HTTP route / Webhook handler 可標 `nodejs` runtime，但邏輯層保持可測試的純函式形式。

### 非同步程式碼規範

- 純函式不引入 async 副作用；非同步行為限制在 route handler / service 層。
- 每個 `Promise` 都需處理錯誤；禁止留下 unhandled rejection。
- 不需要 `await` 的函式不加 `async`。

### TypeScript 嚴格型別安全與抗幻覺規範 (TS Strict Constraints)

1. **嚴格基於 Type/Interface 定義**：
   - 只能使用上下文（Context）中已明確宣告的 `type`、`interface` 或 `class` 所包含的屬性與方法。
   - **嚴禁捏造任何不存在的：物件屬性、可選欄位（Optional Fields）、方法名稱、函式參數與型別、套件 API、第三方服務的 endpoint 或回傳欄位。**

2. **禁止型別系統逃避機制**：
   - 絕對禁止 `as any`、自行捏造的具體型別斷言（`as MyMadeUpType`）。
   - 絕對禁止使用 `!` 非空斷言運算子（`obj!.property`）繞過 null 檢查。
   - 絕對禁止使用 `// @ts-ignore` 或 `// @ts-expect-error` 壓制錯誤。
   - **`unknown` 允許**：外部或不可信輸入可先為 `unknown`，須以 type guard、`in`、`Array.isArray`、`satisfies` 等縮窄後再使用；禁止為跳過檢查而 `as SomeType` 硬轉。
   - 若上下文缺乏特定物件的型別定義，保持 `unknown` 或請使用者提供 interface，嚴禁自行揣摩並填入虛構欄位名稱。

3. **遺失屬性的安全處理機制**：
   - 若程式碼邏輯需要某個屬性，但在現有的 Type 定義中找不到，你**不可以**自行創造它。
   - 你必須改用以下方式處理：
     1. 在程式碼旁加上註解：`// TODO: 請確保實際型別具備此屬性`
     2. 或者使用泛型/佔位符，並在回覆中明確向使用者提出警告。

### Prisma 抗幻覺規範（如有使用 Prisma）

- 撰寫任何 Prisma 查詢前，必須先讀取 `prisma/schema.prisma`，確認 model 名稱、欄位名稱與關聯名稱。
- 嚴禁使用 schema 中未定義的 model、欄位或 relation。
- 只能使用 Prisma Client 官方文件中存在的方法（`findUnique`、`findMany`、`create`、`update`、`delete`、`upsert` 等）。
- model 名稱大小寫須與 schema 完全一致。
- `include` 與 `select` 中只能使用 schema 已定義的關聯名稱；**嚴禁捏造不存在的巢狀關聯**。
- 若欄位或關聯在 schema 中不存在，**不得自行創造**，改用：
  1. 標記 `// TODO: schema 缺少此欄位／關聯，請先 migrate`
  2. 並明確告知使用者需要新增的 schema 定義。
